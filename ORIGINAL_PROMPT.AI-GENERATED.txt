# Research and Design Prompt: A Graph-Theoretic Course Registration System for Middlebury College

## Objective

I am a student at Middlebury College interested in developing a proposal to improve the college's course registration and course scheduling system.

The goal of this project is to investigate how Middlebury currently allocates students to courses and determine whether a system based on **ranked student preferences, graph theory, and optimization algorithms** could produce a more efficient, transparent, and fair outcome.

The first version of this project should **not use utility scores, bidding points, or monetary values**. Students should simply rank the courses they want in order of preference.

The central research question is:

> **Can Middlebury College use a graph-theoretic optimization system to allocate students to courses in a way that improves overall student satisfaction and fairness relative to the current registration system?**

The project should be grounded in Middlebury's actual registration policies, course structure, enrollment capacities, prerequisites, and scheduling system.

# Part II: Translate Registration into a Graph-Theoretic Problem

Develop a formal mathematical representation of the registration system.

## A. Student-Course Bipartite Graph

Represent the system as a bipartite graph.

One set of nodes represents students:

[
S = {s_1, s_2, ..., s_n}
]

The other set represents courses:

[
C = {c_1, c_2, ..., c_m}
]

An edge exists between a student and a course if that student wants to take the course.

Each student submits a strict ranking:

[
c_{i_1} \succ c_{i_2} \succ c_{i_3} \succ ...
]

For example:

Student A:

1. Statistics
2. Algorithms
3. Linear Algebra
4. Spanish

Student B:

1. Algorithms
2. Statistics
3. Economics

The model should initially use only this ranking information.

Do not assign utility points at this stage.

---

## B. Course Capacity Constraints

Each course has a capacity:

[
q(c_j)
]

For example:

* Statistics: 30 seats
* Algorithms: 25 seats
* Linear Algebra: 40 seats

The algorithm must ensure:

[
\text{Number of students assigned to course } c_j \leq q(c_j)
]

No course may exceed its capacity.

---

## C. Student Schedule Constraints

A student should not be assigned to two courses that meet at the same time.

Represent this using a second graph.

Create a **course conflict graph**:

* Each node represents a course.
* An edge connects two courses if they occur at overlapping times.

For example:

```text
Statistics -------- Algorithms
       |
       |
    Economics
```

If Statistics and Algorithms meet at the same time, a student cannot receive both.

The assignment algorithm must therefore satisfy:

[
x_{s,c_1} + x_{s,c_2} \leq 1
]

whenever courses (c_1) and (c_2) conflict.

# Part III: Develop a Ranking-Based Allocation Algorithm

Develop one or more algorithms that assign students to courses based solely on ranked preferences.

The goal is to maximize overall satisfaction while maintaining fairness.

A simple ranking system could assign costs:

| Preference | Cost |
| ---------- | ---- |
| 1st choice | 0    |
| 2nd choice | 1    |
| 3rd choice | 2    |
| 4th choice | 3    |
| 5th choice | 4    |

The optimization problem then attempts to minimize total preference cost:

[
\min \sum_{s \in S}\sum_{c \in C} x_{s,c} \cdot \text{Rank}(s,c)
]

subject to:

1. Course capacity constraints.
2. Schedule conflict constraints.
3. Prerequisite constraints.
4. Student enrollment requirements.
5. Any necessary institutional constraints.

However, do not assume that minimizing total rank is automatically fair.

Compare multiple objectives.

---

# Part IV: Compare Different Definitions of "Fair"

A central goal of the research should be to determine what "fair" actually means.

Compare at least the following models.

## Model 1: Maximize Total Satisfaction

Minimize the total rank assigned across all students.

Example:

```text
Student A: 1st choice
Student B: 1st choice
Student C: 10th choice
```

versus:

```text
Student A: 2nd choice
Student B: 2nd choice
Student C: 2nd choice
```

The second may be considered fairer even if the first produces a better total score.

---

## Model 2: Maximize the Number of First Choices

Maximize:

[
\text{Number of students receiving their first choice}
]

This is simple and intuitive but may create significant inequality.

---

## Model 3: Lexicographic Fairness

First maximize the number of students receiving their first choice.

Then, subject to that, maximize the number receiving their second choice.

Then third choice, and so on.

This creates a hierarchy of objectives.

---

## Model 4: Max-Min Fairness

Attempt to maximize the satisfaction of the least-satisfied student.

The goal is to prevent a small number of students from receiving extremely poor outcomes.

---

## Model 5: Hybrid Objective

Develop a model balancing:

* First-choice allocation.
* Average preference rank.
* Worst-case student outcome.
* Class-year priorities.
* Graduation requirements.

The project should analyze the tradeoffs among these objectives.

---

# Part V: Account for Middlebury-Specific Constraints

The system should not be a generic matching model.

It should reflect real Middlebury conditions.

Consider:

### Prerequisites

A student may only receive a course if they have completed required prerequisites.

### Major and Minor Requirements

Students may have courses that are necessary for graduation or major completion.

The model should determine whether such requirements should receive priority.

### Course Conflicts

Students may rank courses that meet at the same time.

The system must choose among them.

### Multiple Sections

If a course has multiple sections, the algorithm should potentially assign students to different sections.

### Course Sequencing

Some courses may only be offered in particular semesters.

### Seniors Near Graduation

A senior who needs a course to graduate may need priority over a freshman who simply prefers it.

This raises an important ethical and mathematical question:

> Should all preferences be treated equally, or should the model incorporate legitimate institutional priorities?

Analyze both possibilities.

# Part VI: Develop a Course Conflict Graph

One of the most important components of the project should be a separate graph analyzing course demand.

Construct a graph in which:

* Each node represents a course.
* An edge connects two courses if students commonly rank both courses.

The weight of the edge represents the number of students who want both.

For example:

```text
        Statistics
        /         \
       /           \
Algorithms ------ Machine Learning
```

If many students want both Statistics and Algorithms, the edge between those courses is strong.

This graph can identify:

* Courses that should not be scheduled simultaneously.
* Interdisciplinary clusters.
* Courses that compete for the same students.
* Potential demand for new courses.
* Courses that should be scheduled sequentially.
* Departments that should coordinate their timetables.

This creates a second application of graph theory:

> The student-course graph optimizes allocation, while the course-course graph helps optimize future scheduling.

# Part X: Analyze Potential Problems

The research should not assume that optimization is automatically superior.

Analyze important objections.

### Strategic Ranking

Could students manipulate the system by misrepresenting their true preferences?

### Popularity Spirals

Could everyone rank the same courses first, making allocation difficult?

### Institutional Priorities

Should graduation requirements receive priority?

### Inequality

Could a system that maximizes total satisfaction harm a small group of students?

### Transparency

Would students understand why they received or did not receive a course?

### Stability

Would students have incentives to trade or manipulate assignments?

### Computational Complexity

Can the algorithm operate at Middlebury's scale?

### Privacy

How should student preference data be protected?

The final proposal should explicitly address these issues.
